# Docs structure

Как решать, какие `docs/*.md` создавать под конкретный профиль проекта, что писать в каждый.

## Базовая матрица

| # | Файл | Backend REST | Backend RPC | Frontend | CLI | Library | Infra |
|---|------|:---:|:---:|:---:|:---:|:---:|:---:|
| 00 | `README.md` (индекс) | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 01 | `01-overview.md` | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 02 | `02-domain-model.md` | если есть БД | если есть БД | rare | rare | rare | `02-resources.md` (ресурсы IaC) |
| 03 | `03-state-machines.md` | если явные | если явные | если workflow | rare | rare | rare |
| 04 | `04-services.md` (модули) | ✅ | ✅ | `04-components.md` | `04-commands.md` | `04-modules.md` | rare |
| 05 | `05-api.md` | ✅ REST | ✅ RPC/gRPC | внешние API | `05-cli.md` | `05-public-api.md` | ❌ |
| 06 | `06-async-api.md` | если WS/queue | если streaming | если WS/SSE | rare | rare | rare |
| 07 | `07-data-contracts.md` (DTO/schemas) | ✅ | ✅ | `07-types.md` | rare | ✅ | rare |
| 08 | `08-security.md` | если auth | если auth | если auth | rare | rare | ✅ |
| 09 | `09-error-handling.md` | ✅ | ✅ | rare | если есть | если есть | rare |
| 10 | `10-configuration.md` | ✅ | ✅ | ✅ | ✅ | rare | ✅ |
| 11 | `11-known-gaps.md` | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |

Обязательные всегда: `README.md`, `01-overview.md`, `11-known-gaps.md`.

## Что писать в каждом

### `README.md` — индекс

Одна страница-указатель. Только:
- Таблица со ссылками на все созданные файлы (без описания содержимого).
- Раздел «Как читать» — 3-5 сценариев «X → идти в …».
- Одна фраза о соглашениях (пути relative, ссылки `file:line`).

### `01-overview.md` — обзор

- Что за проект — 2-3 фразы.
- Стек — компактная таблица (см. фаза 1 → `stack-detectors.md`).
- Точки входа — файлы `main` / `Application` / `index`.
- Base URL / entry command / API version.
- Ключевые концепции — 5-10 терминов в таблице.
- Роли пользователей — если есть.
- Как запустить локально.
- Внешние артефакты — API-спеки, диаграммы.

### `02-domain-model.md` — сущности и данные

- ER-схема или её текстовое описание.
- Таблица сущностей: поля, типы, связи, ограничения.
- Enum'ы — списком.
- Индексы БД.
- Инварианты и правила валидации.
- Fetch-стратегии / N+1 риски.

### `03-state-machines.md` — переходы состояний

- Одна диаграмма на state machine (ASCII/mermaid).
- Таблица переходов: из → в, триггер, файл:строка.
- Инварианты по состояниям.
- Побочные эффекты переходов (события, публикации).

### `04-services.md` / `04-modules.md` / `04-components.md`

- Карта: модуль → назначение.
- Для каждого — публичные методы/экспорты, побочные эффекты.
- Graph зависимостей (кто кого вызывает).
- Типовые сценарии — sequence вида «A → B → C».

### `05-api.md` / `05-cli.md` / `05-public-api.md`

**REST/GraphQL:** таблица endpoints — метод, путь, роли, DTO, файл. Отдельный раздел для формата ошибок. Ссылка на OpenAPI/schema.

**CLI:** таблица команд — имя, аргументы, флаги, действие. Раздел «Exit codes».

**Library:** экспортируемые модули, стабильность API (public/internal), примеры использования.

### `06-async-api.md`

WebSocket/queues/SSE. Конфигурация, endpoints, авторизация. Payloads. Триггеры отправки. Типичный клиентский flow.

### `07-data-contracts.md` / `07-dto-and-mappers.md` / `07-types.md`

Все контракты обмена данными: DTO/schemas/types, валидации, мапперы entity ↔ contract (если есть).

### `08-security.md`

- Модель аутентификации (JWT/session/OAuth).
- HTTP filter chain / middleware pipeline.
- Публичные vs защищённые маршруты.
- Ролевая матрица.
- CORS / CSRF.
- Секреты и их хранение.

### `09-error-handling.md`

Централизованный обработчик, таблица «исключение → HTTP-код / exit-code / behavior». Кастомные исключения. Формат ошибок.

### `10-configuration.md`

- Конфиг-файлы (properties/yaml/env).
- Environment variables.
- Docker/K8s/CI.
- Команды запуска / сборки / тестов.
- Внешние зависимости и как их поднять локально.

### `11-known-gaps.md`

Живой список:
- Что не реализовано.
- TODO из кода (только значимые).
- Технические долги.
- Security concerns.
- Performance risks.
- Каждый пункт — не описание кода, а описание пропасти.

## Стиль

- **Плотность.** Таблицы > списки > проза.
- **Ссылки.** `path/to/file.ext:LINE` или `path/to/file.ext:START-END`.
- **Русский** (или как выбрано в фазе 2).
- **Без эмодзи и восклицаний.**
- **Заголовки нумерованные:** `## 1. Тема`, `### 1.1 Подтема` — облегчает навигацию.
- **Один раздел = одна тема.** Если получилось >20% файла на одну секцию — вынести в отдельный файл.

## Anti-pattern

- Не пересказывать код словами.
- Не писать «здесь много интересного».
- Не оставлять TODO без ссылки (если это TODO — заводить `T-XXX` через `task-add`).
- Не переиспользовать секции из шаблонов, если содержимое не подходит — лучше пропустить раздел.
